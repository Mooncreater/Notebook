# ③ TCP 三次握手：建立可靠的双向通道

有了 IP 地址，你的电脑需要通过 **TCP（Transmission Control Protocol）** 和服务器建立连接。为什么要三次握手？因为双方都要确认"我能听到你，你也能听到我"。

!!! tip "核心要点"
    TCP 三次握手的本质是**同步双方的序列号（Sequence Number）**。三次交互确定"我发的你能收到，你发的我也能收到"。整个过程需要 1.5 个 RTT（往返时间）。

## 1. 为什么需要 TCP？

IP 协议只负责"尽力而为地"把数据包从 A 送到 B，但它：

- ❌ 不保证送达（可能丢包）
- ❌ 不保证顺序（后发的可能先到）
- ❌ 不保证不重复
- ❌ 不进行流量控制

TCP 在 IP 之上提供了：
- ✅ 可靠传输（确认 + 重传）
- ✅ 顺序保证（序列号）
- ✅ 流量控制（滑动窗口）
- ✅ 拥塞控制（慢启动、拥塞避免）

## 2. 三次握手详细过程

```
客户端 (你的电脑)                          服务器 (Web 服务器)
        │                                      │
        │  ─── 第一次：SYN ──────────────▶  │
        │   SYN=1, seq=x                      │
        │   "在吗？我的初始序列号是 x"          │
        │                                      │
        │  ◀── 第二次：SYN+ACK ───────────  │
        │   SYN=1, ACK=1, seq=y, ack=x+1     │
        │   "在！我的序列号是 y，收到你的 x"    │
        │                                      │
        │  ─── 第三次：ACK ──────────────▶  │
        │   ACK=1, seq=x+1, ack=y+1          │
        │   "收到了！开传！"                    │
        │                                      │
        │       ═══ 连接建立 (ESTABLISHED) ═══  │
```

### 逐个理解每个包

| 次数 | 方向 | 标志位 | 关键信息 | 状态变化 |
|------|------|--------|----------|----------|
| 第 1 次 | 客户端 → 服务器 | SYN=1 | seq=x (客户端初始序列号) | 客户端: CLOSED → SYN_SENT |
| 第 2 次 | 服务器 → 客户端 | SYN=1, ACK=1 | seq=y, ack=x+1 | 服务器: LISTEN → SYN_RCVD |
| 第 3 次 | 客户端 → 服务器 | ACK=1 | seq=x+1, ack=y+1 | 双方: → ESTABLISHED |

### 为什么 ack 是 seq+1？

序列号是**字节流中的位置编号**。SYN 标志本身"消耗"一个序列号（虽然不携带数据）。所以：

- 服务器收到客户端的 SYN(seq=x)，回复 ack=x+1 表示"已收到你的 SYN，期待下一个字节是 x+1"
- 客户端收到服务器的 SYN(seq=y)，回复 ack=y+1 同理

## 3. 为什么不是两次？为什么不是四次？

### 两握手行不行？❌

```
客户端 ── SYN(seq=x) ──▶ 服务器
客户端 ◀── ACK(seq=y, ack=x+1) ── 服务器
```

**问题**：如果第一个 SYN 因网络延迟迟迟未到，客户端超时重发了第二个 SYN。服务器收到两个 SYN 都会回复 ACK，但客户端只会维护一条连接——服务器却以为有两条连接，浪费资源。

> 这就是"历史连接"问题。三次握手的 ACK 让服务器能确认"客户端真的想建立这个连接"。

### 四握手？不需要，合并就行

第二次握手同时携带了 SYN 和 ACK，把"我收到了"和"我也要建立连接"合并成一个包。单独拆开也可以（SYN + 单独的 ACK），但不能更快，反而多一个 RTT。

## 4. TCP 状态机

```
客户端状态：                    服务器状态：

CLOSED                          CLOSED
  │ 主动打开 (connect())          │ 被动打开 (listen())
  ▼                               ▼
SYN_SENT                        LISTEN
  │                               │ 收到 SYN
  │                               ▼
  │ 收到 SYN+ACK               SYN_RCVD
  │ 发送 ACK                      │ 收到 ACK
  ▼                               ▼
  ════════ ESTABLISHED ═══════════
```

!!! tip "日常使用的命令"
    你可以在终端看到系统的 TCP 连接状态：
    ```bash
    netstat -an | findstr ESTABLISHED    # Windows
    netstat -ant | grep ESTABLISHED      # Linux
    ```

## 5. 握手时的性能优化

### TFO (TCP Fast Open)

传统的 TCP 握手：必须完成三次握手才能发数据（1.5 RTT 白白浪费）。

TFO 的改进：在 SYN 包中直接携带数据。

```
传统：SYN → SYN+ACK → ACK → 数据 (1.5 RTT 后才发数据)
TFO： SYN(data) → SYN+ACK → ACK+data (0 RTT 即可发数据！)
```

前提：客户端之前已经和服务器建立过连接，缓存了一个 **TFO Cookie**。

### 握手期间还发生了什么

三次握手期间，TCP 协议栈还在协商：
- **MSS（Maximum Segment Size）**：单次 TCP 报文最大数据量，通常为 MTU - 40 (IP头) - 20 (TCP头) = 1460 字节
- **窗口缩放因子（Window Scale）**：允许窗口大小超过 65535 字节
- **SACK（Selective ACK）**：选择性确认，只重传丢失的段

## 6. 连接建立后的世界

三次握手完成后，TCP 连接进入 ESTABLISHED 状态。此时：

1. 双方各自维护了对方的序列号
2. 拥塞窗口（cwnd）初始化为一个 MSS，进入**慢启动**阶段
3. 数据传输可以开始了——但这只是"裸"的 TCP，对于 HTTPS，还需要在 TCP 之上再穿一层 TLS 加密外衣

!!! danger "三次握手中的安全攻击"
    **SYN Flood 攻击**：攻击者发送大量 SYN 但不回复最后的 ACK，导致服务器大量连接卡在 SYN_RCVD 状态，耗尽资源。防御手段：SYN Cookie（服务器不在握手阶段分配资源，而是把状态信息编码进 seq 返回，只有收到合法 ACK 才分配资源）。

```
DNS 返回 IP: 110.242.68.66:443
        │
        ├─ 客户端随机生成初始序列号 seq=x
        │
        ├─ SYN(seq=x) ─────────────────▶ 服务器
        │     客户端状态 → SYN_SENT
        │
        ├─ ◀─────────────── SYN+ACK(seq=y, ack=x+1)
        │     服务器状态 → SYN_RCVD
        │
        ├─ ACK(ack=y+1) ───────────────▶ 服务器
        │     双方状态 → ESTABLISHED
        │
        └─ TCP 通道就绪 → 但 HTTPS 还需要 TLS → 下一阶段
```
