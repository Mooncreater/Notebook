# ⑥ 服务器处理：从网卡到应用层

HTTP 请求终于到达了服务器。和客户端一样，服务器的内核协议栈也要做一遍 TCP/IP 的解封装。之后数据才会交付给真正的 Web 服务进程。

!!! tip "核心要点"
    服务器端的处理是"镜像的逆过程"：物理信号 → 数据帧 → IP 包 → TCP 段 → TLS 解密 → HTTP 请求。真正的业务处理可能涉及 CDN、反向代理、负载均衡、应用服务器、数据库等多层架构。

## 1. 数据到达服务器：协议栈解封装

```
客户端 ──── 层层封装 ────▶ 网络 ────▶ 服务器端层层解封装

                    ┌─────────────────────────┐
 应用层 (HTTP)      │ GET / HTTP/1.1           │ ← 服务进程收到的
                    ├─────────────────────────┤
 TLS 层             │ 解密后的 HTTP 明文        │ ← TLS 在此解密
                    ├─────────────────────────┤
 传输层 (TCP)       │ 排序、去重、确认           │ ← 内核 TCP 栈
                    ├─────────────────────────┤
 网络层 (IP)        │ 路由到本机                │
                    ├─────────────────────────┤
 链路层 (MAC)       │ 确认 MAC 地址是发给我的    │
                    ├─────────────────────────┤
 物理层             │ 电信号转二进制             │ ← 网卡收到的
                    └─────────────────────────┘
```

!!! tip "网卡中断与 DMA"
    网卡收到数据后通过**硬件中断**通知 CPU，然后用 **DMA（直接内存访问）** 把数据直接写入内核缓冲区，不需要 CPU 逐字节拷贝。这就是为什么千兆网卡不会把 CPU 跑满。

## 2. 现代服务器的典型架构

一个大型网站的服务器端不是你想象中的"一台机器跑一个程序"，而是一个庞大的分层架构：

```
你 (浏览器)
  │
  ▼
┌──────────────────────────────────────┐
│ CDN 边缘节点          ← 静态资源（图片/CSS/JS/HTML缓存） │
│ (如果命中缓存，到这里就结束了！)      │
└──────────────┬───────────────────────┘
               │ (未命中或动态请求)
               ▼
┌──────────────────────────────────────┐
│ DNS 智能解析          ← GSLB 把用户路由到最近的机房     │
└──────────────┬───────────────────────┘
               ▼
┌──────────────────────────────────────┐
│ 负载均衡器 (LB)       ← Nginx/LVS/F5，分发到后端服务器  │
└──────────────┬───────────────────────┘
               ▼
┌──────────────────────────────────────┐
│ 反向代理              ← Nginx/Envoy，TLS 卸载、路由      │
└──────────────┬───────────────────────┘
               ▼
┌──────────────────────────────────────┐
│ Web 应用服务器        ← Tomcat/Node.js/Gunicorn/Go      │
└──────────────┬───────────────────────┘
               ▼
┌──────────────────────────────────────┐
│ 数据库 / 缓存 / MQ    ← MySQL + Redis + Kafka 等         │
└──────────────────────────────────────┘
```

## 3. 逐层拆解

### CDN（内容分发网络）

CDN 是离用户最近的"缓存服务器"，全国/全球部署节点。

```
北京用户访问 www.taobao.com
    │
    ├─ DNS 返回北京 CDN 节点的 IP（而非杭州淘宝源站）
    │
    └─ 北京 CDN：图片/CSS/JS 有缓存 → 直接返回 ✅
                           HTML 没缓存 → 回源到杭州源站 → 缓存后返回
```

> CDN 能扛住 99% 的流量。淘宝首页几十亿 PV，如果没有 CDN，源站早就被打爆了。

### 负载均衡

当请求到达源站机房，首先遇到负载均衡器：

| 类型 | 代表 | 工作层 | 特点 |
|------|------|--------|------|
| **DNS 负载均衡** | GSLB | 应用层 | 按地理位置分配，粒度粗 |
| **四层 LB** | LVS、F5 | 传输层 (TCP) | 只看 IP+端口，性能高 |
| **七层 LB** | Nginx、HAProxy | 应用层 (HTTP) | 可按 URL/Cookie 路由，灵活 |

常见负载均衡算法：

```
轮询 (Round Robin)    → 依次分配，简单但可能不均衡
加权轮询              → 好机器多分配
最少连接 (Least Conn) → 哪台机器空闲给哪台
IP 哈希               → 同一 IP 始终打到同一台（Session 保持）
一致性哈希            → 扩容时迁移最少数据
```

### 反向代理（Nginx 的角度）

反向代理是客户端和服务器的中间人：

```
              请求 → Nginx → 应用服务器
客户端 ──▶              │
              响应 ← Nginx ← 应用服务器
```

Nginx 的典型职责：
1. **TLS 卸载**：Nginx 负责 HTTPS 加密解密，后端用 HTTP（省 CPU）
2. **静态资源**：图片/CSS/JS 由 Nginx 直接返回，不惊动应用层
3. **缓存**：可配置反向代理缓存，减少对后端的请求
4. **限流**：防止单个 IP 刷爆服务
5. **压缩**：gzip/brotli 压缩响应体

### 应用服务器

这是你写的业务代码真正运行的地方：

- **Java**：Spring Boot 跑在 Tomcat / Netty 上
- **Python**：Django / Flask 跑在 Gunicorn / uWSGI 上
- **Node.js**：Express / Koa / Fastify（自带 HTTP 服务器）
- **Go**：net/http + Gin / Echo 等框架

以访问淘宝首页为例，后端可能的处理流程：

```
1. 收到 GET / HTTP/1.1
2. 检查 Session/Cookie 确认用户身份
3. 调用推荐服务 → 获取个性化商品推荐
4. 调用商品服务 → 获取首页展示的商品列表
5. 调用广告服务 → 获取竞价广告位
6. 调用用户服务 → 获取未读消息数
7. 组装 HTML 模板（服务端渲染 SSR）
8. 返回完整 HTML
```

> 这就是所谓的"微服务"架构：首页这一个请求，后端可能调用了十几个微服务。

### 数据库与缓存层

```
应用服务器
    │
    ├─ Redis (缓存) ── 命中？ → 直接返回，~1ms
    │      └─ 未命中 ↓
    │
    ├─ MySQL (数据库) ── 查询，~5-50ms
    │      └─ 结果写回 Redis 缓存
    │
    └─ Kafka / MQ ── 异步消息（如埋点、日志）
```

## 4. Todo：从一台机器到一座机房

!!! tip "一个真实的请求链路"
    当淘宝首页被访问时，实际的服务器端链路可能是：
    ```
    CDN → LVS (四层) → Nginx 集群 (七层) → Spring Boot (业务)
    → 推荐服务 → 商品服务 → 广告服务 → 用户服务
    → Redis (缓存) → MySQL (数据库)
    → 组装 HTML → 返回
    ```
    这背后可能有几十台服务器协同工作，而用户只感觉到"页面一下子就出来了"。

## 5. 服务器响应往回传

服务器的响应经历了和请求镜像的过程：

```
应用服务器生成 HTML
    │
    ├─ HTTP 响应头 + Body
    │
    ├─ (如果 Nginx 前面做了 TLS 卸载) Nginx 对响应加密
    │
    ├─ TCP 分段 → IP 分包 → 链路层成帧 → 物理层
    │
    └─ 数据包一跳一跳回到客户端
```

## 小结

```
HTTP 请求到达服务器
        │
        ├─ CDN 边缘节点 ──── 静态资源缓存命中 → 直接返回 ✅ (最快路径)
        │
        ├─ 负载均衡器 ──── 分发到某台 Nginx
        │
        ├─ Nginx ──── TLS 解密 / 静态资源 / 限流
        │
        ├─ 应用服务器 ──── 执行业务逻辑
        │      ├─ Redis 查缓存
        │      ├─ MySQL 查数据库
        │      └─ 调用其他微服务 (RPC)
        │
        ├─ 组装 HTML 响应
        │
        └─ 响应原路返回 → 经 TLS 加密 → TCP → IP → 网络 → 客户端 → 下一阶段
```
