# 04 · RSA 非对称加密 — 锁和钥匙不是同一把

---

## 一、这是什么？

前面学的 AES 是**对称加密**：加密解密同一把钥匙。

**RSA 是非对称加密：两把钥匙，一把锁🔓（公钥），一把开🔑（私钥）。**

> **公钥加密，私钥解密。公钥随便给，私钥死也不能给。**

---

## 二、一句话理解

```
          公钥（锁）               私钥（钥匙）
            🔓                      🔑
             │                       │
  任何人拿来锁数据 ──→ 密文 ──→ 只有钥匙能打开

反过来也行：
  我用私钥签名 ──→ 签名 ──→ 任何人用公钥验证，确认是我签的
```

**两个经典场景：**

| 场景 | 谁加密 | 谁解密 | 解决什么问题 |
|------|--------|--------|-------------|
| **加密通信** | 别人用我的公钥锁 | 我用私钥开 | 只有我能看 |
| **数字签名** | 我用私钥签名 | 别人用公钥验 | 证明是我发的 |

---

## 三、为什么牛逼？—— 解决 AES 的致命问题

### AES 的问题：钥匙怎么给？

你和百度通信，想用 AES：
1. 你生成一把 AES 密钥
2. 你得把这把密钥**发给百度**——但你还没有加密通道！
3. 密钥在网络上裸奔 → 中间人可以偷走密钥 → AES 白费

这叫**密钥分发问题**。

### RSA 怎么解决？

```
1. 百度把公钥贴网上（公钥 = 锁，谁都能看）
2. 你拿到百度的公钥
3. 你随机生成一把 AES 密钥
4. 用百度的公钥把 AES 密钥锁起来 → 发给百度
5. 百度用私钥解开 → 拿到 AES 密钥
6. 双方用这把 AES 密钥加密后续通信
```

**RSA 只用来传 AES 的钥匙，传完之后切回 AES（因为 RSA 太慢）。**

## 四、RSA 的数学原理（手把手，纯数字）

RSA 依赖一个简单事实：**两个大质数相乘很容易，但把乘积拆回两个质数极难。**

### 你来生成一对 RSA 密钥

#### Step 1：选两个质数
```
p = 3
q = 11
n = p × q = 33
```
（真实 RSA 用几百位的质数，这里用小的方便算）

#### Step 2：计算 φ(n)（欧拉函数）
```
φ(n) = (p-1) × (q-1) = 2 × 10 = 20
```

#### Step 3：选公钥指数 e
e 必须和 φ(n)=20 互质（最大公约数为1）
```
选 e = 3  （3 和 20 互质 ✓）
```

#### Step 4：计算私钥指数 d
d 满足：e × d ≡ 1 (mod φ(n))
即 3 × d ≡ 1 (mod 20)
→ d = 7  （因为 3 × 7 = 21，21 mod 20 = 1 ✓）

#### Step 5：密钥对出炉
```
公钥：(n=33, e=3)     ← 可以公开
私钥：(n=33, d=7)     ← 藏好！
```

### 加密一个数字

假设明文 M = 4：

```
密文 C = M^e mod n
       = 4^3 mod 33
       = 64 mod 33
       = 31
```

### 解密

```
明文 M = C^d mod n
       = 31^7 mod 33
       = 27512614111 mod 33
       = 4  ✅ 回来了！
```

## 五、真实 RSA 和玩具 RSA 的区别

| | 玩具 RSA | 真实 RSA |
|------|---------|----------|
| p, q 大小 | 3 和 11 | 几百位的质数 |
| n 大小 | 33（6位二进制） | 2048位或4096位 |
| 破解难度 | 口算 | 宇宙寿命算不完 |
| 加密内容 | 只能加密数字 | 配合填充方案加密任意数据 |

> 真实 RSA 不会直接加密消息，而是配合 OAEP 填充方案，然后加密。

## 六、RSA 和 AES 的关系 —— 混合加密

HTTPS 实际用的是 **RSA + AES 混用**：

```
① RSA 登场：客户端用服务器公钥加密"会话密钥" → 发过去
② 服务器用私钥解开 → 双方都有了"会话密钥"
③ AES 登场：双方用"会话密钥"进行 AES 对称加密通信
④ 会话结束 → "会话密钥"丢弃
```

**RSA：跑第一棒（传密钥）→ AES：跑完剩下的（加解密数据）**

为什么？RSA 比 AES 慢几百倍，不适合加密大量数据。



## 七、RSA 的另一个方向：数字签名

加密是"公钥锁，私钥开"。
签名反过来——**私钥签名，公钥验证**。

### 签名流程

```
① 我对消息做 SHA-256 哈希 → 得到"摘要"
② 用我的私钥加密摘要 → 得到"签名"
③ 消息 + 签名一起发给你
④ 你用我的公钥解密签名 → 得到摘要1
⑤ 你对消息自己做 SHA-256 → 得到摘要2
⑥ 摘要1 == 摘要2 → 签名有效 ✅
```

### 签名证明了什么？

- **消息是我发的**（我的私钥只有我有）
- **消息没被篡改**（改一个字，哈希就变了）

## 八、RSA 在现实中的身影

| 场景 | RSA 干什么 |
|------|-----------|
| **HTTPS** | 握手阶段，传 AES 密钥 |
| **SSH 登录** | 你的公钥存在服务器上，用私钥登录 |
| **JWT Token** | RS256 签名，防篡改 |
| **Git 签名** | `git commit -S` 用私钥签名提交 |
| **GPG 加密邮件** | 用对方公钥加密，对方私钥解密 |

## 九、练习题

### 练习 1：手工 RSA

用上面玩具RSA的参数：
- 公钥 (n=33, e=3)
- 私钥 (n=33, d=7)

加密数字 7，密文是多少？

<details><summary>答案</summary>

```
C = 7^3 mod 33 = 343 mod 33 = 13
```
密文 = 13

解密验证：
```
M = 13^7 mod 33 = 62748517 mod 33 = 7 ✅
```

</details>

### 练习 2：概念辨析

以下场景用的是 RSA 加密还是 RSA 签名？

a) 你用 GitHub 配置了 SSH Key
b) 你发一封 GPG 加密邮件
c) `git commit -S`
d) 浏览器访问 `https://baidu.com`

<details><summary>答案</summary>

a) SSH：服务器存你的公钥，你用私钥证明身份 → **签名**
b) GPG 加密邮件：用收件人公钥加密 → **加密**
c) `git commit -S`：用私钥签名提交 → **签名**
d) HTTPS：RSA 传 AES 密钥 → **加密**（传密钥阶段）

</details>
