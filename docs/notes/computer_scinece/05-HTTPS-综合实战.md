# 05 · HTTPS 综合实战 — 所有知识大串联

---

## 一、你打开 `https://baidu.com`，完整流程

这一章把前面学的**DNS、证书、RSA、AES、Base64**全部串起来。

你敲下回车后，一共经历 **5 大阶段**：

```
① DNS 解析      →  找到百度的 IP
② TCP 连接      →  和百度建立"电话线"
③ TLS 握手      →  双方验身份 + 协商加密方式（核心！）
④ 加密通信      →  AES 加密聊天
⑤ 连接关闭      →  挂断
```

---

## 二、阶段 ①：DNS 解析（知识点：01-DNS、02-Base64）

你敲 `baidu.com`，浏览器不认识，要找 IP：

```
浏览器检查缓存（0ms）
  → 没命中
操作系统缓存（1ms）
  → 没命中
hosts 文件
  → 没命中
问路由器/ISP DNS（10-50ms）
  → 返回 110.242.68.66
```

现在你拿到了百度的 IP：`110.242.68.66`。

---

## 三、阶段 ②：TCP 三次握手（建立"电话线"）

```
你  ──SYN──→  百度     "喂，能听到吗？"
你  ←─SYN+ACK─  百度   "能，你听得到我吗？"
你  ──ACK──→  百度     "能，开始说话吧"
```

现在一根可靠的"电话线"已经建立。

---

## 四、阶段 ③：TLS 握手 —— 最核心的部分

TLS 握手要完成三件事：
1. **验身份**：确认对面真的是百度（不是冒充的）
2. **协商加密方式**：你们俩用哪种加密算法
3. **生成会话密钥**：这把钥匙只在这通"电话"里用，挂断就扔

### ③-1：Client Hello（你先开口）

你告诉百度：
```
"你好，我支持的加密套件有：
  - TLS_AES_256_GCM_SHA384
  - TLS_AES_128_GCM_SHA256
  - ...
  我生成了一个随机数 R1 = 8A3F...（28字节）"
```

### ③-2：Server Hello（百度回应）

百度回复：
```
"我选 TLS_AES_256_GCM_SHA384（挑你支持的最安全的那个）
 这是我的随机数 R2 = B2C1...（28字节）"
```

### ③-3：Certificate（百度掏身份证）

百度掏出它的证书：
```
{
  "我是": "www.baidu.com",
  "谁证明": "DigiCert（全球知名 CA）",
  "我的公钥": "0xA3F2B1C4...（百度 RSA 公钥）",
  "有效期": "2025-01-01 ~ 2026-01-01",
  "指纹": "SHA-256: 8A3F..."
}
```

### ③-4：你验身份证

你的浏览器顺着证书链往上查：
```
百度的证书 ← 谁签的？ ← DigiCert
DigiCert 的证书 ← 谁签的？ ← DigiCert 根证书
DigiCert 根证书 ← 在浏览器的信任名单里吗？ ← ✅ 是
→ 证书合法！对面真是百度！
```

**这就是 DNS 投毒为啥对 HTTPS 无效：假服务器拿不出真证书。**

### ③-5：Key Exchange（传 AES 密钥）

你生成一个"预主密钥"（Premaster Secret），用百度的公钥加密：

```
premaster = 我自己随机生成的 48 字节
encrypted_premaster = RSA加密(premaster, 百度公钥)
→ 发给百度
```

百度用私钥解开，拿到 premaster。

**现在双方都有：R1 + R2 + premaster**

双方用同样的公式算出"会话密钥"：
```
会话密钥 = SHA-256(R1 + R2 + premaster)
```

**这把会话密钥才是真正用来 AES 加密聊天的。**

---

## 五、阶段 ④：AES 加密通信

双方都有了同一把"会话密钥"，开始 AES-GCM 加密通信：

```
你：AES加密("GET / HTTP/1.1", 会话密钥) → 百度
百度：AES加密("<html>百度首页</html>", 会话密钥) → 你
你：AES加密("Cookie: SESSION_ID=abc123", 会话密钥) → 百度
...
```

**RSA 已经下班了，现在是 AES 在干活。**

---

## 六、阶段 ⑤：挂断

通信结束，双方扔掉"会话密钥"。下次连接重新来一遍 TLS 握手，生成全新的会话密钥。

## 八、实战：用浏览器看真实的 TLS 握手

### 打开 F12 看证书

1. 按 F12 → 点 "Security"（安全）标签
2. 访问 `https://baidu.com`
3. 点 "View certificate" → 你能看到：
   - 颁发给：www.baidu.com
   - 颁发者：DigiCert 之类的 CA
   - 有效期
   - 公钥信息（2048位 RSA 或 ECC）

### 用 openssl 模拟 TLS 连接

```bash
openssl s_client -connect baidu.com:443 -showcerts
```

你会看到：
1. 服务器证书（PEM 格式，就是 Base64 包裹的）→ 知识点：**Base64**
2. 证书链 → 知识点：**CA、证书**
3. 协商的加密套件 → 知识点：**AES、RSA**

输出示例：
```
---
Certificate chain
 0 s:CN = *.baidu.com
   i:C = BE, O = GlobalSign nv-sa, CN = GlobalSign RSA OV SSL CA 2018
-----BEGIN CERTIFICATE-----
MIIG6DCCBdCgAwIBAgIQBx0...
（这就是 Base64 编码的证书内容！）
-----END CERTIFICATE-----
---
SSL handshake has read 4856 bytes and written 451 bytes
New, TLSv1.3, Cipher is TLS_AES_256_GCM_SHA384
```

## 九、常见问题

### Q：为什么 HTTPS 不用纯 RSA 加密全部数据？
RSA 太慢了。加密 1KB 数据，RSA 比 AES 慢几百到上千倍。
所以只用 RSA 传一小段"会话密钥"（几十字节），然后 AES 干重活。

### Q：如果 CA 被黑了会怎样？
CA 被黑 = 公安局被假冒 → 攻击者能签发假证书 → 中间人攻击可行。
历史上发生过（DigiNotar 2011 年被黑 → 倒闭）。
所以现在用 **证书透明度（Certificate Transparency）**：所有证书公开记录，异常签发能被发现。

### Q：HTTPS 防什么？不防什么？

| 防 | 不防 |
|----|------|
| ✅ 中间人偷看内容 | ❌ 你电脑被种木马（键盘记录器） |
| ✅ 中间人篡改内容 | ❌ DNS 泄露访问了哪些域名（SNI 明文） |
| ✅ 冒充网站（假证书通不过） | ❌ 服务器本身被黑 |
| ✅ 重放攻击 | ❌ 你知道你在和百度聊天，但不知道百度是不是已经被黑了 |
